# zk-prover-wasm — browser prover for sealed gifts (frontend integration)

Compiles the `zk-circuits` withdraw circuit to wasm so a Groth16 proof can be
generated **in the recipient's browser** — the note secret never leaves the
client. This crate's phase job was to answer the make-or-break UX question
before any UI was built: *is client-side proving fast and small enough?*

## Answer: yes ✅ (measured)

| Metric | Value |
|---|---|
| Circuit constraints | 2,457 |
| Proving key (compressed) | **770 KB** (uncompressed 1.5 MB) |
| Wasm module | **468 KB** |
| Verifying key | 536 bytes |
| Proof size | 192 bytes (Groth16, fixed) |
| Native prove time | ~228 ms |
| **Browser/Node prove time** | **~2.9 s** (single-threaded wasm) |
| **Browser proof verified on-chain** | ✅ **`true`** on the deployed verifier |

A ~770 KB one-time (cacheable) key download and a ~3 s proof with a progress
indicator is a perfectly workable withdraw UX. Crucially, a proof generated by
the wasm module **verifies against the deployed testnet verifier**
(`CBL4G6ER…ML43`) — the whole stack works browser → chain.

## Surface

- `note_commitment_hex(secret)` → the Merkle-leaf commitment to deposit.
- `prove_withdraw_js(pk, secret, siblings, bits, recipient)` → JSON
  `{a,b,c,root,nullifier}` in the pool contract's uncompressed format.

The frontend wraps these in `frontend/src/lib/zkProver.ts` (lazy-loads wasm + key)
and `frontend/src/lib/shieldedNote.ts` (note secrets + claim-link encoding).

## Build / measure

```bash
cargo test --release measure_pk_size_and_prove_time -- --nocapture   # native numbers
cargo test --release emit_wasm_fixture -- --ignored                  # pk.bin + vk.json
wasm-pack build --target web --release --out-dir ../frontend/src/wasm/zkprover
node measure.mjs            # real wasm prove time (nodejs target)
node prove_and_emit.mjs     # emit a browser proof, then verify it on-chain:
# stellar contract invoke --id CBL4G6ER..ML43 ... -- verify --vk .. --proof .. --signals ..
```

## ⚠️ Slice-grade

The proving key comes from a **test-RNG trusted setup** and must match the vk the
pool was initialized with. Production needs a real setup ceremony, the finalized
Poseidon2 constants, and an audit (see `zk-circuits` and `contracts/shielded-pool`
READMEs). `pkg/`, `pkg-fixture/`, and generated keys are build artifacts.
